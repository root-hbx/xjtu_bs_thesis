\BiChapter{相关技术介绍}{}

\BiSection{3GPP NTN 协议与协议栈}{}

非地面网络 (Non-Terrestrial Network, NTN) 是 3GPP 将蜂窝移动通信协议扩展到卫星等非地面节点的重要标准方向。
与传统地面蜂窝网络相比，NTN 的核心变化在于：无线接入链路不再发生在固定地面基站和近距离终端之间，而是发生在高速运动的卫星或高空平台与地面用户设备之间。
这使得传播时延、多普勒频移、可见性窗口等成为协议设计中必须显式处理的因素。
3GPP 已在 Release 17 中引入 NR NTN 的基础能力，并在后续 Release 18 和 Release 19 中继续完善移动性管理等业务支持\cite{3gppNtn,threegppRel17,threegppRel18,threegppRel19}。

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{dtc_arch.pdf}
\caption{手机直连卫星网络架构图} \label{dtc_arch}
\end{figure}

从网络架构看，NTN 大体分为两种模式，如图~\ref{dtc_arch}所示：

\begin{itemize}
    \item 透明载荷模式（TRANS）。卫星主要作为射频转发节点，gNB 协议处理仍位于地面网关侧。
    这一模式更强调长距离接入链路、馈电链路和地面基站处理过程。
    \item 再生载荷模式（REGEN）。卫星具备更完整的基带处理能力，可以在星上运行 gNB 相关功能。
    通过星间链路（ISLs，Inter-Satellite Links）相互连接，并基于馈电链路（Feed Links）连接地面核心网。
    这一模式要求仿真系统能够表达星上 gNB、用户终端和核心网之间的协议交互。
\end{itemize}

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{ntn_protocol_stack.pdf}
\caption{3GPP NTN 标准网络协议栈} \label{ntn_protocol_stack}
\end{figure}

从协议栈角度看，NTN 仍然沿用 5G NR 的分层设计，
包括 PHY、MAC、RLC、PDCP、RRC、NAS 以及核心网控制与用户面协议，如图~\ref{ntn_protocol_stack}所示。
不同之处在于，NTN 需要在这些协议流程中引入卫星运动带来的时间和频率补偿。
例如，UE 需要根据广播的星历信息和服务小区配置进行上行定时调整；随机接入过程需要适应更长传播时延；
RRC 和移动性管理需要处理更快变化的小区可见性与切换条件。
SIB19、Timing Advance、多普勒预补偿等机制，都是 NTN 相比地面 NR 更加特殊的部分。
本文后续关注的 StarPike 系统也正是围绕这些 MAC 及以上层可观察到的 NTN 行为进行大规模仿真。

\BiSection{现有低轨卫星轨道仿真平台}{}

低轨卫星网络研究通常需要首先获得卫星位置、速度、可见性和链路状态等时变信息。
公开的两行轨道根数 (Two-Line Element, TLE) 数据可以描述卫星轨道状态，
Skyfield 等工具可以基于 SGP4 模型从 TLE 推导卫星在不同时刻的位置和速度\cite{skyfield,celestrak}。
在此基础上，研究者可以计算卫星与地面站或用户终端之间的距离、仰角、传播时延、多普勒频移和可见性窗口，从而驱动后续网络仿真或协议实验。

围绕低轨星座网络，已有工作大体可以分为模拟 (simulation) 和仿真 (emulation) 两类。
StarPerf，Hypatia 和 ns-3-leo 属于模拟类平台，它们通过抽象模型描述星座拓扑、链路状态和网络协议行为，
适合快速扫描大规模星座配置、端到端路径和网络性能\cite{hypatia,ns3leo,starperf}。
模拟方法的优势是轻量、可扩展、便于重复实验，但其节点和协议通常是模型化的，并不运行真实协议栈。

StarryNet 和 OpenSN 属于仿真类平台，强调以容器、网络命名空间或系统组件运行更接近真实系统的软件实体，
用于研究天地一体网络和低轨卫星网络中的路由、拥塞控制和端到端性能\cite{starrynet,opensn}。
仿真方法的优势是系统行为更接近真实部署，能够复用更多实际软件栈；代价是资源开销更高，规模扩展更困难。

值得注意的是，现有低轨卫星网络平台通常将无线接入层抽象为链路或网络层状态，
重点关注 IP 层以上的路径选择、队列、拥塞控制和路由收敛\cite{hypatia,starrynet,opensn}。
对于手机直连卫星网络而言，许多关键现象发生在蜂窝接入层，例如 RACH 竞争、RRC 状态迁移、NTN 定时补偿、SIB19 广播和切换信令。
仅依靠传统星座网络仿真平台，难以观察这些 3GPP NTN 协议流程本身。
因此，本文工作并不是替代现有星座网络平台，而是在其轨道和链路建模能力之上，进一步补足大规模 NTN 无线接入协议仿真的能力。

\BiSection{现有地面移动网络协议栈}{}

OpenAirInterface 是一个开源蜂窝网络协议栈项目，提供 4G/5G RAN 的软件实现，
并支持 gNB、UE、软调制解调器、RRC、MAC、RLC、PDCP 等模块\cite{oai}。
Open5GS 是一个开源移动核心网项目，提供 AMF、SMF、UPF 等 5G 核心网功能\cite{open5gs}。
二者组合后，可以构建较完整的端到端 5G 实验环境：OAI 负责无线接入网侧协议和 UE/gNB 行为，Open5GS 负责核心网注册、会话管理和用户面转发。
对于 NTN 研究而言，这类开源协议栈的优势在于可修改、可复现，并且能够运行真实的 RRC、NAS 和用户面流程。

在蜂窝 RAN 内部，nFAPI 和 F1 是两个重要的模块边界。nFAPI 定义了 PHY 与 MAC 之间的接口，MAC 并不直接消费原始 I/Q 采样，
而是通过接口获取上行接收结果、信道质量、调度相关指示和传输块解码状态等信息。
F1 接口则用于分离集中单元 (Central Unit, CU) 和分布单元 (Distributed Unit, DU)，其中 F1-C 承载控制面信令，F1-U 承载用户面数据。
借助这些标准边界，可以在不完全改写上层协议栈的情况下，对 PHY 或 CU/DU 功能部署进行模拟、替换和重组。

本文提出的 StarPike 正是利用这些协议边界来降低大规模 NTN 仿真的计算开销。
对于 NTN 研究问题而言，MAC 及以上层并不需要逐样本恢复完整物理层波形，
而是需要在正确时间获得等价的信道质量、解码结果、Timing Advance、调度偏移和广播信息。
因此，可以在 nFAPI 边界下使用轻量模拟模型产生上层所需的观测结果，
同时保留 OAI/Open5GS 中原有的 RRC、NAS、PDU 会话和核心网流程。这一设计为后续 PHY-Lite 和集中式星历引擎提供了技术基础。

\BiSection{蜂窝协议测试床与商用测试平台}{}

蜂窝协议测试床通常依赖专用硬件或商用测试平台来运行真实空口信号和协议流程。
例如，基于 USRP 等软件无线电设备的测试床可以在可控环境中发送和接收真实射频信号，从而观察调制解调、信道估计、随机接入和调度等底层行为。
商用平台如 Amarisoft 和 Rohde \& Schwarz 的测试系统则在稳定性、协议覆盖和工程支持方面更成熟，
常用于设备测试、基站验证和标准一致性实验\cite{amarisoft,rohdeNtn}。

这些硬件测试床的优势是物理层和协议流程保真度高，能够真实反映射频链路、空口调度和终端接入过程。
但是，它们通常面向单链路或小规模实验，难以直接扩展到数百个 UE--gNB 对并发的低轨 NTN 场景。
此外，商用测试平台成本高且闭源，扩展能力受限于射频硬件数量、频谱环境和实验部署复杂度。

因此，大规模手机直连卫星网络研究需要在协议保真度和系统可扩展性之间取得平衡。
完全基于真实 PHY 的测试床适合小规模高保真验证，
而星座级 NTN 研究更需要一种能够保留 MAC 及以上层协议行为、同时降低底层计算开销的仿真系统。
本文后续章节将围绕这一目标展开，说明 StarPike 如何在已有开源协议栈和标准接口基础上构建面向大规模手机直连卫星网络的实验平台。
