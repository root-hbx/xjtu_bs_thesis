\BiChapter{StarPike 系统实现}{}

\BiSection{原型系统概述}{}

本章介绍 StarPike 原型系统的具体实现，关注各模块如何落地，
包括实验配置读取、星历参数生成、共享内存同步、PHY-Lite 与 OAI 的接口适配，以及容器化运行流程。
原型系统的实现目标是尽量复用现有开源 NTN 协议栈，在最小化改动 MAC 及以上层代码的前提下，
将高开销的重复计算和完整 PHY 处理移出运行时节点内部。

在部署方式上，每个 UE 和 gNB 仍然可以作为独立容器或进程运行，
从而保留接近真实系统的协议栈隔离和日志观测方式。
与朴素全栈仿真不同，容器内部不再独立完成完整星历推导和波形级 PHY 处理，
而是通过共享内存读取 CEE 发布的当前时隙参数，并使用 PHY-Lite 将这些参数转化为 nFAPI 侧事件。

\BiSection{集中式星历引擎实现}{}

CEE 的实现可以分为配置解析、轨道传播、链路预算、NTN 参数生成和共享内存发布五个步骤。
配置解析阶段读取卫星 TLE、gNB 载荷标识、UE 地理位置、实验起始时间、仿真持续时间和链路预算参数。
轨道传播阶段基于 SGP4 计算每颗卫星在每个时间点的位置和速度。
链路预算阶段根据卫星与 UE 的几何关系计算斜距、仰角、传播损耗和接收信噪比。
NTN 参数生成阶段进一步推导传播时延、多普勒频移、Timing Advance、调度偏移和 SIB19 兼容字段。
最后，共享内存发布阶段将每个 UE--gNB 对的参数向量写入环形缓冲区，供 UE、gNB 和 PHY-Lite 读取。

CEE 在每个毫秒级时隙 $t_k$ 为每个 UE $u_i$ 与 gNB $g_j$ 生成参数向量 $\vec{p}_{ij}(t_k)$。
表\ref{tab_cee_fields} 给出了原型系统中主要字段及其用途。

为了避免各节点读取到不同时间轴上的状态，CEE 使用统一的墙上时间作为全局推进基准。
系统启动时，CEE 记录实验起始墙上时间 $T_0$，并将第 $k$ 个时隙映射为 $T_0+k$ ms。
在运行过程中，CEE 始终按照墙上时间推进写指针，而不是等待某个慢速容器读取完成。
这一选择与 StarPike 的设计目标一致，先前在~\ref{problem_statement}中给出了定量分析：
相较于让全局时间被个别滞后节点拖慢，保持整体时间轴稳定对 NTN 协议实验更重要。

\BiSection{共享内存环形缓冲区}{}

CEE 需要向大量 UE/gNB 进程高频发布参数。
如果使用套接字逐节点发送，会产生大量系统调用和消息复制；如果使用文件轮询，则无法满足毫秒级更新需求。
因此，StarPike 原型使用 POSIX 共享内存实现单写多读的环形缓冲区。
CEE 是唯一写者，所有 UE、gNB 容器节点都是读者。具体算法伪代码如~\ref{pr_code}所示：

\begin{algorithm}[H]
    \caption{CEE 参数发布与节点读取流程}
    \label{pr_code}
    \KwIn{实验配置、当前墙上时间、环形缓冲区}
    \KwOut{每个 UE--gNB 对在当前时隙的 NTN 参数}
    \ForEach{time slot $t_k$ in lookahead window}{
        \ForEach{UE--gNB pair $(u_i,g_j)$}{
            compute $\vec{p}_{ij}(t_k)$ from orbit and link model\;
            set slot sequence to odd\;
            write $\vec{p}_{ij}(t_k)$ into shared memory\;
            set slot sequence to even\;
        }
    }
    \ForEach{reader request at slot $t_k$}{
        read sequence $s_1$\;
        copy parameter vector $\vec{p}_{ij}(t_k)$\;
        read sequence $s_2$\;
        accept the vector if $s_1=s_2$ and $s_1$ is even\;
    }
\end{algorithm}

环形缓冲区按照“时隙编号 + UE--gNB 对编号”索引。
每个槽位保存一个参数向量和序列计数器。
CEE 写入某个槽位时，先将序列计数器设为奇数，表示该槽位正在更新；
写入完整参数向量后，再将计数器设置为偶数，表示该槽位可被一致读取。
读者读取时，先读一次计数器，再复制参数向量，最后再次读取计数器。
只有当两次计数器相同且为偶数时，读者才接受该结果；否则重新读取。
这一 seqlock 同步适合 StarPike 的访问模式：写者只有一个，而读者很多，
且读者偶发重试的代价远低于全局阻塞。

CEE 还维护一个前瞻窗口 $W$。
在墙上时间到达 $t_k$ 之前，CEE 预先计算并写入未来 $W$ ms 的参数。
这样即使某些 UE/gNB 进程在调度上有轻微抖动，它们也能从共享内存中读取到当前或未来时隙的参数，而不必等待 CEE 实时计算。
当某个读者因严重落后而占据时序槽位时，StarPike 允许覆盖旧数据。
这是一个有意的取舍：系统优先保持全局墙上时间推进，而不是为了保留某个落后节点的历史状态拖慢所有节点。

\BiSection{PHY-Lite 与 OAI 接口适配}{}

PHY-Lite 的实现重点是将 CEE 提供的链路状态转换为 OAI MAC 及以上层能够消费的事件。
在原生 OAI 中，MAC 层通过 PHY 回调和 nFAPI 类接口获得上行接收结果、CQI、CRC、HARQ 反馈、随机接入指示和调度相关信息。
StarPike 保留这些接口形态，但将信息来源从完整 I/Q 波形处理替换为 CEE 参数和统计 PHY 模型。

PHY-Lite 的核心数据路径包括三个步骤。
第一步是信道质量映射。
对于当前服务 UE--gNB 实例，PHY-Lite 从共享内存读取 $SNR_{ij}(t_k)$
，并按照 NR/NTN 中 CQI 上报的含义映射为 CQI 序列。
MAC 层随后仍然基于 CQI 进行 MCS 和资源分配选择。
第二步是传输块结果判定。
PHY-Lite 根据当前 MCS、资源分配和 $SNR_{ij}(t_k)$ 查询或计算对应的 BLER，
并用随机采样或阈值判定生成 CRC 成功/失败结果。
第三步是定时与 NTN 事件注入。
PHY-Lite 将 $TA_{ij}$、调度偏移 $K_{ij}$、可见性 $v_{ij}$ 和 SIB19 相关字段转化为 MAC/RRC 可见的定时观测、系统信息可达状态和随机接入结果。

这种实现方式保留了 OAI 上层协议的执行路径。
RRC 建链、NAS 注册、PDU 会话建立、RLC/PDCP 处理和核心网交互仍然由原有代码执行；PHY-Lite 只替换原本由软调制产生的底层观测。
因此，当链路 SNR 下降、可见性消失或 TA 不满足要求时，上层仍会通过真实协议流程表现出 RACH 失败、RRC 释放或用户面性能下降。

\BiSubsection{广播与系统信息接收}{}

在 NTN 接入中，UE 需要完成小区搜索、同步、系统消息接收和 NTN 相关参数获取。
原生全栈实现通常需要通过 PHY 解析同步信号波形。
在 StarPike 中，CEE 已经生成了当前 gNB 的星历、TA、调度偏移、服务窗口和可见性信息，
PHY-Lite 因此直接向上层暴露系统信息可用事件。
对于 UE 而言，它仍然经历系统信息获取流程；区别在于系统信息内容由 CEE 填充，
并通过 PHY-Lite 注入，而不是由 UE 独立从波形中恢复。

为了保持 NTN 语义，PHY-Lite 并不会对 UE 宣布广播接收结果。
只有当 CEE 判断 UE 与 gNB 在当前时隙可见，且链路质量满足系统消息接收要求时，
PHY-Lite 才向 MAC/RRC 报告系统信息可用。
如果仰角过低、传播损耗过大或服务窗口结束，UE 将反馈系统信息不可达或测量质量下降，
从而触发与真实 NTN 接入一致的上层行为。

\BiSubsection{随机接入与连接建立}{}

随机接入过程是 PHY-Lite 保真度的关键。
StarPike 保留 NR/NTN 的随机接入状态机，包括前导发送、RAR 接收、Msg3 发送和竞争解决等步骤。
不同之处在于，前导检测、RAR 是否成功、Msg3 解码结果等事件由 PHY-Lite 根据 CEE 参数和统计模型生成。

具体而言，UE 发起随机接入前，PHY-Lite 从 CEE 读取当前服务 gNB 的传播时延、TA、调度偏移和 SNR。
如果当前可见性为真且链路质量满足门限，则 PHY-Lite 在正确的 MAC 时序上生成前导检测和 RAR 可达事件；
否则产生失败或超时。
Msg3 和后续下行控制消息的成功概率由 SNR--CQI--BLER 模型决定。
随机接入结果仍受卫星运动、链路质量和 NTN 定时关系影响，而简化完整波形级检测链路。

\BiSubsection{NAS 与用户面传输}{}

RRC 连接建立后，后续 NAS 注册、PDU 会话和用户面数据转发仍然由 OAI/Open5GS 原有流程执行。
PHY-Lite 对这些流程的影响主要体现在传输块成功率和时延。
当 CEE 输出的 SNR 较高且可见性稳定时，PHY-Lite 将生成较高的解码成功率，吞吐量和 RTT 保持稳定。
当卫星运动导致链路质量下降或发生切换时，PHY-Lite 会增加传输块失败、HARQ 或 RLC 重传概率，
在上层表现为吞吐下降和时延抖动。

StarPike 不是简单地在应用层注入流量延迟：
用户面数据仍然穿过 PDCP、RLC、MAC、GTP-U 和核心网用户面路径，只是底层 PHY 观测由轻量模拟生成。
因此，研究者仍然可以观察协议栈内部的队列积压、重传、会话状态和端到端性能变化。

\BiSection{容器化部署与运行流程}{}

StarPike 的实验运行由管理脚本统一编排。
管理脚本首先读取实验配置，生成 UE、gNB、核心网等全局模块所需的配置文件。
随后创建容器或进程组，为每个实体分配网络命名空间、IP 地址、日志目录和共享内存访问权限。
在所有容器启动前，CEE 会先完成初始前瞻窗口的参数预计算，确保协议栈运行时能够立即读取到 NTN 状态。

一次完整实验的启动顺序如下：
首先启动 Open5GS 核心网组件，确保 AMF、SMF 和 UPF 就绪。
其次启动 CEE，使全局星座状态开始按墙上时间推进。
然后启动 gNB 实例，加载 PHY-Lite 适配层并连接核心网。
最后启动 UE 实例，使其通过 PHY-Lite 获取系统信息、执行随机接入、完成 RRC/NAS 建链并建立 PDU 会话。
实验运行过程中，后台管理脚本持续收集各容器日志、CEE 状态、RACH/RRC 事件、吞吐和 RTT 等指标。

为便于调试，StarPike 为每个模块保留独立日志。
CEE 日志记录每个时隙的全局推进状态、前瞻窗口长度和共享内存覆盖情况；
PHY-Lite 日志记录 CQI、BLER、CRC 结果、随机接入事件和系统信息可达状态；
OAI/Open5GS 日志则用于验证 RRC、NAS、PDU 会话和核心网行为。
这些日志共同支撑第六章中的实验评估。
