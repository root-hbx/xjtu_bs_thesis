\BiChapter{绪论}{Introduction}

\BiSection{研究背景及意义}{Research Background and Significance}

\BiSubsection{手机直连卫星网络的发展背景}{}

随着低轨卫星星座和第五代移动通信技术的发展，天地一体化网络正在从专用卫星终端场景逐步走向普通移动终端场景。传统低轨卫星互联网通常依赖相控阵天线等专用用户终端，
而手机直连卫星网络 (Direct-to-Cell Network, DTCN) 则希望让卫星直接为标准或仅经过少量修改的商用手机提供连接能力。
Starlink、AST SpaceMobile、Iridium 等企业已经开始部署或测试相关服务，监管机构和运营商也在持续推动卫星手机直连卫星能力的商用化\cite{starlinkMobile,astSpaceMobile,iridiumNtnDirect}。
这一趋势使低轨卫星系统不再只是面向固定终端或专用终端的补充网络，而是逐渐成为未来移动通信基础设施的重要组成部分。

在协议体系上，手机直连卫星网络可以大致分为三类：
早期面向卫星电话的专有协议、经过卫星适配的 LTE/NR 私有协议栈，以及由 3GPP 标准化的非地面网络 (Non-Terrestrial Network, NTN) 协议栈。
前两类方案要么面向专用卫星电话，难以兼容普通商用手机；要么由厂商私有实现主导，开放性和可复现实验能力有限。
相比之下，3GPP NTN 是目前唯一开放且标准化的蜂窝卫星协议方向，已经在 Release 17、Release 18 和 Release 19 中持续演进\cite{3gppNtn,threegppRel17,threegppRel18,threegppRel19}。
因此，围绕 3GPP NTN 构建可复现、可扩展的实验平台，对于研究下一代手机直连卫星网络具有重要意义。

3GPP NTN 支持透明载荷和再生载荷两类部署模式：
透明载荷模式下，卫星仅作为弯管转发节点，调度、无线资源控制和移动性管理等协议处理仍然由地面基站完成；
再生载荷模式下，卫星具备完整的基带处理能力，并可以通过星间链路互联，作为空间中的 gNB 参与无线接入和移动性管理。
因此，手机直连卫星网络具有高速运动、传播时延大、链路质量快速变化和覆盖波束持续移动等特点。
特别是在低轨场景下，卫星相对地面的高速运动会导致传播时延、Doppler Shift（多普勒频移）、TA（时间提前量）、小区可见性在毫秒级持续变化。
这些动态因素会直接影响随机接入、RRC 连接、NAS 注册、PDU 会话建立，以及上层的网络传输性能。

\BiSubsection{星座级 NTN 仿真的必要性}{}

近年来，手机直连卫星产业部署规模的持续高速增长。
以 Starlink 为例，截至 2026 年 3 月，其已经有超过 10000 颗卫星入轨，其中 650 余颗具备手机直连卫星能力，并已经在多个大洲提供数据、视频和消息等服务；
其后续还计划将 DTC 星座规模扩展至 15000 颗卫星\cite{starlinkMobile,starlinkDtcScale}。
这说明手机直连卫星网络已经不再是少量链路的功能验证问题，而是具有百星级、千星级甚至更大规模部署特征的系统问题。
单一 UE--gNB 实例的链路测试只能验证基本协议流程，无法覆盖真实星座级部署中出现的规模级现象。
此外，许多 NTN 问题只有在规模化场景下才会暴露：
例如，低轨卫星波束边界扫过地面时，覆盖区内大量 UE 可能几乎同时触发小区重选或条件切换，进而造成 RACH 竞争等现象。
再生载荷和星间链路的引入，还会使路由、切换和核心网功能部署变成星座级协同问题，其结果取决于全局拓扑和快速变化的空间状态。
因此，针对上述问题的研究，需要研究者具备能够运行大规模 NTN 协议栈的实验能力。

\BiSubsection{现有软件仿真的时钟漂移问题}{}

现有开源组件为构建 NTN 实验环境提供了基础。例如，可以使用 Skyfield 和公开 TLE 数据生成卫星轨道状态\cite{skyfield,celestrak}，
使用 OpenAirInterface (OAI) 和 Open5GS 分别实例化无线接入网和核心网协议栈\cite{oai,open5gs}。
本文将物理世界线性流逝的时间记为墙上时间 $t_w$，将被仿真节点感知到的逻辑时间记为虚拟时间 $t_v$。
理想情况下应满足 $t_v = t_w$，从而保证协议栈按照真实时间推进。
但随着仿真节点数量增加，有限主机资源需要同时承载大量软调制解调、信道计算、协议处理和容器通信任务，导致 $t_v$ 逐渐落后于 $t_w$，形成时钟漂移 $\delta(t)=|t_v-t_w|$。
对于普通 IP 网络仿真而言，有限时间漂移可能只影响性能统计；但对于 NTN 接入过程而言，RACH、TA、调度偏移、HARQ 和 RRC 定时器都依赖严格实时约束，时钟漂移会直接破坏协议正确性。

基于上述现有开源组件的一种直接思路，是将轨道轨迹生成工具与完整 NTN 软件协议栈进行朴素组合：
由 Skyfield 根据 TLE 生成卫星运动轨迹，再由 OAI NTN 和 Open5GS 分别运行无线接入网与核心网协议流程。
本文首先在 Dell PowerEdge R740 高性能服务器上构建了这一朴素软件仿真环境，并观察到：
当 UE 和 gNB 节点总数增至26时，系统已接近资源耗尽，并出现显著的虚拟时间滞后。
600 秒虚拟时间在 20 UE-gNB 实例链路下，需要约 750 秒墙上时间；在 26 实例场景中则需要超过 1100 秒墙上时间。
随之而来的是信令层和网络层的连锁退化：RAR 失败和 RRC 释放次数显著增加，随机接入过程超过 3GPP 规定的重传阈值；
下行吞吐从单链路时约 30 Mbps 下降到多链路场景下的 1 Mbps 以下，平均端到端 RTT 则从约 50 ms 上升到甚至超过 1000 ms。
这表明时钟漂移不仅是平台性能问题，更会改变被研究协议本身的可观测行为。

造成这一问题的根源在于：星座级 NTN 仿真需求和传统全栈软件仿真结构之间的矛盾。
首先，传播时延、多普勒频移、TA 和可见性等 NTN 状态本身随 UE--gNB 实例、卫星轨道不断时变。
CEE 的本质并不是假设不同实例共享相同轨迹，而是将轨道与 NTN 状态提前集中式计算，
在运行阶段把原本由函数调用完成的状态转移，转换为对已生成状态表的读值，从时间角度对状态计算进行前置和负载均衡。
其次，完整物理层波形处理占据了主要 CPU 开销，而 NTN 协议研究真正关心的是 MAC 及以上层可观察到的信道质量、解码结果和定时信息。
这些因素共同导致主机资源被重复状态推导和低层信号处理消耗，真正需要按时执行的 NTN 控制流程反而得不到足够计算资源。

基于上述观察，本文提出 StarPike：一个面向大规模手机直连卫星场景下的 NTN 协议仿真系统。
其目标是在保持 3GPP NTN 协议状态机和 MAC 及以上层行为可观测性的前提下，大幅降低时钟漂移，实现大规模手机直连卫星场景的 NTN 协议仿真。
StarPike 的核心意义在于，它并不简单地牺牲协议真实性来换取规模，
而是从 NTN 仿真中的状态计算和物理层开销两个关键瓶颈入手，对执行模型进行重组织。
通过该系统，可以在普通服务器上支持数百规模的 UE--gNB 实例，
并为未来研究手机直连卫星网络中的随机接入、移动性管理、O-RAN 功能拆分等问题提供实验基础。

\BiSection{国内外研究应用现状}{}

围绕手机直连卫星网络的研究大体可分为三类：
NTN 协议与机制研究、低轨星座网络仿真与仿真平台、蜂窝协议测试床与商用测试工具。

\BiSubsection{NTN 协议与手机直连卫星网络研究现状}{}

3GPP NTN 将地面 NR 协议扩展到卫星环境，是当前手机直连卫星网络研究的重要标准基础\cite{3gppNtn}。
已有研究围绕 TA 补偿、HARQ、RLC ARQ、低轨切换和卫星场景下的无线接入机制展开，分析了大传播时延、高多普勒和快速移动覆盖对蜂窝协议的影响\cite{ntnTaCompensation,ntnHarqRlc,leoHandoverSolutions}。
近年来，随着手机直连卫星网络进入产业部署阶段，研究者也开始关注其体系结构优化设计、位置隐私、无卫星导航接入和星上核心网功能部署等问题\cite{democratizingDtc,dtcLocationLeakage,dtcWithoutGnss,spaceCore}。

这些工作揭示了 NTN 协议和架构中的关键问题，但其中相当一部分实验依赖硬件测试床、信道模拟器或小规模链路级验证。
此类方法能够提供较高物理层保真度，却难以扩展到数十、数百甚至更多 UE 和卫星链路并发的场景。
因此，现有协议研究亟需一种能够同时保留 3GPP 接入流程和星座级动态的可扩展实验基座。

\BiSubsection{低轨星座网络仿真平台}{}

Hypatia、StarPerf、StarryNet、OpenSN等低轨卫星网络仿真平台\cite{hypatia,starperf,starrynet,opensn}
已为星座拓扑、星间路由、拥塞控制和端到端性能分析提供了重要工具。
这些平台通常关注 IP 层及以上行为，能够研究星座拓扑变化、瓶颈链路识别、路由收敛等网络问题\cite{misleadingLeoCc,leocc,leoBottleneck}。

然而，这类平台的抽象边界通常停留在无线接入层之上，不执行完整的蜂窝 RAN 协议过程。
换言之，它们能够较好地研究星间路由和 IP 层传输，却无法直接观察 RACH、HARQ、RRC 状态迁移、SIB19 广播、TA 更新和 NTN 切换信令等接入网行为。
对于手机直连卫星网络而言，这些接入层事件往往是性能退化和系统失效的直接原因。
因此，本文工作与低轨星座网络仿真平台是互补关系：
后者解决星座拓扑和网络层问题，StarPike 则聚焦缺失的大规模 NTN 无线接入实验能力。

\BiSubsection{蜂窝协议测试床与 NTN 网络协议栈}{}

在蜂窝协议实现方面，OpenAirInterface、srsRAN 和 Open5GS 等开源项目为 4G/5G RAN 和核心网实验提供了可复现基础\cite{oai,srsran,open5gs}。
商用平台如 Amarisoft 和 Rohde \& Schwarz 的 NTN 测试方案则能提供较高稳定性和工程成熟度\cite{amarisoft,rohdeNtn}。
这些系统的优势在于协议栈真实度高，可以运行实际 RRC、NAS、MAC 和核心网流程，适合进行功能验证、单链路测试和硬件在环实验。

但无论是开源软件栈还是商用测试设备，当前仍面向单链路或小规模场景。
商用平台成本高、闭源且难以横向扩展；开源软件栈虽然可修改，
但若直接以完整软调制解调器形式运行大量节点，会快速受到 CPU、内存和实时调度能力限制。
本文所关注的问题并非重新实现一个蜂窝协议栈，而是在继承现有真实协议栈保真度的基础上，
重构其在大规模 NTN 场景下的执行方式，使其能够支持星座级实验。

\BiSection{本文主要内容}{}

本文提出并实现 StarPike，一个面向大规模手机直连卫星场景下的 NTN 协议仿真系统。
支持在有界时钟漂移的约束下开展 3GPP NTN 协议过程的规模化实验。

具体而言，本文的主要内容包括：
\begin{enumerate}[wide, label=\arabic*),  labelindent=21pt]
    \item 定量分析大规模手机直连卫星场景下的 NTN 协议仿真中时钟漂移问题。
    本文首先基于 OpenAirInterface、Open5GS、Skyfield 和公开 TLE 数据构建朴素 NTN 仿真环境，
    量化其在规模上升时出现的虚拟时间滞后、随机接入失败、RRC 释放、吞吐下降和 RTT 膨胀等现象，并系统分析其根本原因。
    \item 设计集中式星历引擎 (Centralized Ephemeris Engine, CEE)。CEE 统一读取星座轨迹和 UE/gNB 配置，
    在毫秒级时间粒度上集中推导 UE--gNB 实例的传播时延、多普勒频移、SNR、TA、调度偏移等信息。
    通过集中式预计算，将运行时的 NTN 状态转移从函数调用转换为状态读值，从时间维度实现大规模仿真的计算负载均衡。
    \item 设计基于模拟的 NTN 轻量物理层 PHY-Lite。PHY-Lite 以 nFAPI 等 3GPP 标准边界为基础，
    用轻量模拟替代完整波形级 PHY 执行，保留 MAC、RLC、PDCP、RRC、NAS 和核心网流程。
    其核心是根据 CEE 提供的链路状态生成上层所需的 CQI、解码结果、定时观测和随机接入事件，
    从而降低 CPU 开销并保持 MAC 及以上层行为一致性。
    \item 实现 StarPike 原型系统并进行实验评估。
    本文在容器化实验环境中集成 OAI、Open5GS、Skyfield/SGP4、共享内存环形缓冲区和 PHY-Lite 模型，
    并通过微基准实验验证其扩展性、运行时开销和协议行为保真度。
    实验表明，StarPike 在相同硬件条件下，能够支持 UE--gNB 实例规模至1000节点开展NTN协议仿真，
    并保持关键 NTN 信令和传输行为与高保真参考实现的一致性。
\end{enumerate}

\BiSection{本文组织结构}{}

具体章节安排如下：

\begin{enumerate}
    \item 绪论。本章介绍手机直连卫星网络的研究背景、星座级协议仿真的必要性、现有工作的不足，以及本文提出的 StarPike 协议仿真系统。
    \item 相关技术介绍。本章将介绍 3GPP NTN 网络协议、低轨卫星网络、开源移动网络协议栈、共享内存同步，以及物理层模拟建模等后续章节所需的基础技术。
    \item 大规模手机直连卫星场景下的时钟漂移定量分析。本章将构建朴素 NTN 仿真环境，分析其在规模增加时的时钟漂移、信令失败和网络性能退化现象，并归纳大规模手机直连卫星网络仿真的关键瓶颈。
    \item StarPike 系统设计。本章将给出 StarPike 总体架构，介绍集中式星历引擎和基于模拟的 NTN 轻量物理层，以及它们如何协同约束时钟漂移并保持协议保真度。
    \item StarPike 系统实现。本章将描述原型系统的工程实现，包括容器化部署、共享内存环形缓冲区和 NTN 轻量物理层实现。
    \item 实验设计与结果。本章将展示 StarPike 的实验设置和评估结果，从可扩展性、CPU 开销、时钟漂移、NTN 信令信息正确性这四个方面验证系统。
    \item 结论与展望。本章总结全文工作，并讨论系统在物理层细粒度建模、星间路由联动、核心网功能星上部署等后续扩展方向。
\end{enumerate}
